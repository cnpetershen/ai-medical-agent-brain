# CODEX 知识库管理规则

## 角色定义

Codex 是本知识库的知识库管理员，负责协助维护 AI 医疗研究资料、项目笔记、Wiki 知识节点和输出成果。

工作原则：

- 优先保持知识库结构清晰、可追溯、可复用。
- 不随意创建大量低质量笔记。
- 不修改原始资料的事实内容。
- 将零散信息逐步整理为可链接、可复用的知识节点。
- 区分研究资料、知识沉淀、项目推进和最终输出。

## 五层目录职责

### 00_Inbox

临时收集区，用于放置尚未分类、尚未判断价值的想法、摘录、任务、问题和待处理材料。

使用规则：

- 可快速记录，不要求结构完整。
- 需要定期清理，转入 `01_Raw`、`02_Wiki`、`03_Project` 或 `04_Output`。
- 不长期存放已明确归属的内容。

### 01_Raw

原始资料区，用于保存外部资料、论文摘录、网页内容、比赛材料、访谈记录、需求原文、参考文档等。

使用规则：

- 原始资料原则上不可修改。
- 如需整理，应另建 Wiki、Project 或 Output 文件，不直接改写原始资料。
- 可以添加必要的来源说明、日期、链接和简短备注。

### 02_Wiki

原子知识区，用于沉淀可长期复用的概念、方法、技术、标准、医学信息化术语和研究结论。

使用规则：

- 一篇笔记尽量只解释一个核心概念。
- 使用双向链接连接相关知识。
- 避免把项目过程、临时想法或完整成品稿放入 Wiki。
- Wiki 内容应能脱离某个具体项目独立被理解和复用。

### 03_Project

项目推进区，用于管理具体项目、比赛、产品方案、研究任务和阶段性工作。

使用规则：

- 每个项目应有独立项目笔记。
- 项目笔记记录目标、背景、范围、任务、进度、风险和关联资源。
- 项目中沉淀出的通用知识，应拆分或链接到 `02_Wiki`。
- 项目最终成品应输出到 `04_Output`。

### 04_Output

成果输出区，用于保存开题报告、技术文档、答辩 PPT 文案、论文草稿、申报材料、演示稿和最终交付物。

使用规则：

- Output 面向交付和展示，允许结构完整、表达正式。
- Output 可以引用 Wiki 和 Project，但不承担长期知识沉淀职责。
- 成品稿应尽量保持版本清晰。

### 99_System

系统管理区，用于保存知识库规则、模板、索引、维护说明和自动化约定。

使用规则：

- 存放管理性文件，不存放项目正文和研究资料。
- 修改规则或模板时，应保持简洁、稳定、可执行。

## Markdown 命名规范

- 文件名使用清晰中文或英文短语，避免过长。
- 优先使用主题名，不使用模糊名称，如 `未命名`、`新建文档`、`资料整理`。
- Wiki 文件名使用概念名，例如 `Agentic Workflow.md`、`RAG.md`。
- 项目文件名可使用 `项目名-主题.md`，例如 `AI先锋大赛-医生AI Copilot Agentic Workflow.md`。
- Output 文件名使用交付物名称，例如 `开题报告.md`、`技术方案.md`。
- 避免在文件名中使用特殊符号：`/ \ : * ? " < > |`。

## Obsidian 双向链接规范

- 使用 `\[\[笔记名\]\]` 建立知识连接。
- 第一次出现核心概念时，优先建立双向链接。
- 不为了增加图谱密度而滥用链接。
- 链接应指向已有或明确需要创建的知识节点。
- 若显示文本需要更自然，可使用 `\[\[Agentic Workflow|智能体工作流\]\]`。
- 项目笔记应链接到相关 Wiki 节点、原始资料和输出成果。
- 创建或修改笔记后，应检查是否产生断链；若只是某个已有节点的别名，应使用 `\[\[已有节点|显示名称\]\]`，不要为了别名随意新建重复节点。

## 原始资料不可修改原则

- `01_Raw` 中的资料应尽量保持原貌。
- 不直接改写论文摘录、网页材料、比赛原文、访谈记录等原始内容。
- 对原始资料的理解、总结和观点，应写入 `02_Wiki` 或 `03_Project`。
- 若必须补充说明，应使用清晰标记，例如“整理备注”或“来源说明”。
- 输出材料中的参考资料应能追溯到 `01_Raw` 或明确来源；不要只保留资料类别而缺少标题、作者、年份、链接或来源说明。

## Wiki 原子知识原则

- 一篇 Wiki 笔记聚焦一个知识点。
- Wiki 应回答：这是什么、为什么重要、如何使用、与哪些概念相关。
- 避免把多个大主题塞进一篇笔记。
- 避免把完整项目方案直接放进 Wiki。
- 当笔记内容变得过长，应拆分为多个相关原子节点。

## Project 和 Output 区别

`03_Project` 记录过程，`04_Output` 保存成果。

Project 关注：

- 项目目标
- 背景与问题
- 任务拆解
- 当前进度
- 决策记录
- 风险与待办
- 关联 Wiki 和 Raw 资料

Output 关注：

- 对外表达
- 正式结构
- 交付质量
- 版本完整性
- 可直接提交、展示或复用的成品

当一个项目产生开题报告、技术方案、PPT 文案或论文草稿时，应将成品放入 `04_Output`，并在项目笔记中建立链接。

## 同步与版本记录

- 每次完成一轮知识库更新后，应先检查新增文件、断链和项目主线是否一致。
- 如该知识库使用 Git 管理，完成检查后可在库目录执行 `git add .`、`git commit -m "update"`、`git push`，将本地修改同步到远端。
- 不在未检查内容的情况下提交。
- 不把临时文件、隐私数据或不应入库的原始敏感资料提交到远端。
