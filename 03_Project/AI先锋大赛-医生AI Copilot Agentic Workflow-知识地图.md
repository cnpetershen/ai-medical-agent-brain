# 医生AI Copilot Agentic Workflow-知识地图

## 文件定位

本文用于规划“医生AI Copilot Agentic Workflow”项目需要沉淀的知识体系。

所有知识点必须服务于项目定位：打造贯穿医生诊前、诊中、诊后的连续照护闭环。

本文只规划知识节点，不改变项目方向，不替代项目核心定义文件。

优先级说明：

- P0：项目主线必需，优先沉淀。
- P1：支撑方案完整性，应尽快沉淀。
- P2：增强工程、论证或答辩表达，可后续沉淀。

## A. 项目核心概念

|知识点|为什么需要|服务项目哪部分|优先级|
|---|---|---|---|
|医生AI Copilot|定义本项目的核心产品形态，明确 AI 是医生工作流程中的协同助手，而不是替代医生的自动诊断系统。|项目总定位、用户角色、项目边界|P0|
|Agentic Workflow|解释为什么采用“智能体 + 可控工作流”的方式组织医疗任务，避免完全自主 Agent 偏离临床流程。|核心架构、三大 Workflow 调度|P0|
|连续照护闭环|说明诊前、诊中、诊后之间如何形成连续服务链条，是项目价值主线。|项目定位、核心目标、Output 表达|P0|
|医疗Copilot Agent|定义智能调度中枢的职责，包括理解任务、组织上下文、调度 Workflow 和辅助医生决策。|核心架构、技术方案|P0|
|医生主导的人机协同|明确医生是核心使用者和决策者，AI 负责辅助信息处理与流程管理。|项目边界、风险控制、答辩论证|P0|
|可控与可追溯|医疗场景需要可审核、可解释、可回溯，防止 AI 输出成为黑箱。|医疗安全、技术方案、评估指标|P1|

## B. 三大 Workflow 知识

|知识点|为什么需要|服务项目哪部分|优先级|
|---|---|---|---|
|诊前信息采集 Workflow|用于说明患者基础信息、症状信息、历史资料如何在就诊前被整理为医生可用的信息。|诊前环节、信息获取、患者参与|P0|
|诊中辅助决策 Workflow|用于说明 AI 如何在医生诊疗过程中辅助知识查询、检查结果整理和病历摘要，但不替代医生判断。|诊中环节、临床辅助决策|P0|
|诊后医嘱执行 Workflow|用于说明医嘱、用药、康复任务和随访如何形成可跟踪的诊后管理流程。|诊后环节、连续照护闭环|P0|
|患者信息结构化|诊前采集需要将自由文本、问卷、历史记录整理为结构化信息，降低医生阅读负担。|诊前信息采集 Workflow|P1|
|检查结果整理|诊中需要整合 LIS、PACS 或模拟检查数据，辅助医生快速理解患者状态。|诊中辅助决策 Workflow|P1|
|医嘱执行跟踪|诊后闭环需要知道患者是否执行医嘱，并触发提醒或随访任务。|诊后医嘱执行 Workflow|P1|
|随访任务生成|连续照护不能止于一次就诊，需要把风险、复诊和健康管理转化为后续任务。|诊后管理、连续照护闭环|P1|

## C. AI 技术支撑

|知识点|为什么需要|服务项目哪部分|优先级|
|---|---|---|---|
|Agent|解释医疗 Copilot Agent 的基本能力，包括任务理解、上下文组织和动作调度。|智能调度中枢、技术方案|P0|
|RAG|用于支持医疗知识、指南、病历资料和项目资料的检索增强，减少凭空生成。|诊中辅助决策、知识查询|P0|
|Memory|用于维护患者上下文、医生偏好、流程状态和历史任务记录，支撑连续照护。|诊前到诊后的跨阶段衔接|P0|
|风险画像|围绕单个患者维护血压控制、依从性、症状变化和连续照护断点等关注优先级。|Patient Memory、诊后反馈回流、医生复诊关注点|P0|
|Tool Calling|用于调用数据查询、任务生成、提醒、随访管理等工具，让 Agent 能执行具体动作。|三大 Workflow 执行|P0|
|Human-in-the-loop|医疗场景必须保留医生审核与确认环节，防止 AI 自动越权。|项目边界、风险控制|P1|
|Prompt / 指令约束|用于约束 Agent 输出格式、行为边界和医疗安全表达。|技术方案、演示系统|P1|
|Guardrails|用于限制不合规、不安全或超出边界的 AI 行为。|医疗安全、项目边界|P1|

## D. 医疗业务支撑

|知识点|为什么需要|服务项目哪部分|优先级|
|---|---|---|---|
|HIS|理解医院核心业务系统的角色，明确本项目不是 HIS 替代平台，而是围绕医生流程做辅助。|项目边界、医疗数据接口|P0|
|EMR|电子病历是医生诊疗信息的重要来源，需要支持病历摘要、历史记录理解和上下文组织。|诊前信息整理、诊中辅助决策|P0|
|LIS|检验数据是诊中判断的重要材料，需要支持检查结果整理和异常提示。|诊中辅助决策 Workflow|P1|
|PACS|影像系统是医疗数据来源之一，本项目可引用影像报告或模拟影像结论，不做影像诊断系统。|诊中辅助决策、项目边界|P1|
|医疗数据中台|用于理解多源医疗数据如何汇聚、治理和供 Workflow 使用。|医疗数据接口、工程实现|P1|
|患者健康数据|诊后连续照护需要患者侧反馈，如血压、服药、康复任务完成情况。|诊后医嘱执行 Workflow|P1|
|医疗隐私与最小必要原则|患者数据使用必须有边界，支撑项目安全性和答辩可信度。|风险控制、项目约束|P0|

## E. 工程实现

|知识点|为什么需要|服务项目哪部分|优先级|
|---|---|---|---|
|Workflow Orchestration|用于说明三大 Workflow 如何被调度、串联、触发和管理。|核心架构、工程方案|P0|
|数据接口|用于定义 Agent 和 Workflow 如何读取 HIS、EMR、LIS、PACS 或模拟数据。|医疗数据接入、技术方案|P0|
|模拟数据|医疗真实数据涉及隐私，比赛或方案验证阶段需要使用模拟数据完成演示和评估。|方案验证、Output 材料|P0|
|评估指标|用于衡量方案是否降低医生负担、提升连续性和提高流程执行质量。|项目论证、答辩、技术方案|P0|
|任务状态管理|Workflow 需要记录任务是否完成、是否需要医生审核、是否进入下一环节。|三大 Workflow 执行|P1|
|日志与追溯|医疗 AI 辅助过程需要记录输入、输出、调用工具和医生确认结果。|可追溯、风险控制|P1|
|原型系统架构|用于把项目从概念方案落到可演示系统，包括前端、Agent 服务、数据层和工具层。|技术文档、演示系统|P1|
|权限与角色控制|医生和患者的数据权限不同，需要体现医疗场景的角色边界。|用户角色、隐私保护|P2|
|异常处理机制|当数据缺失、工具失败、AI 输出不确定时，需要进入人工确认或降级流程。|医疗安全、Workflow 稳定性|P2|
