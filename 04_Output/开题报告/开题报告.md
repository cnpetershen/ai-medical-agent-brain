# 医生AI Copilot Agentic Workflow 开题报告

## 一、课题名称

医生AI Copilot Agentic Workflow：面向诊前、诊中、诊后连续照护闭环的医疗智能助手研究与设计

## 二、课题背景

医疗服务流程中，医生需要在有限时间内完成患者信息理解、病历与检查资料查阅、医学知识检索、诊疗记录整理、医嘱生成和诊后随访管理。现有医院信息系统通常以 [[HIS]]、[[EMR]]、[[LIS]]、[[PACS]] 等业务系统为中心，能够支撑数据记录和业务运转，但在跨阶段信息整合、诊后执行反馈、复诊准备和医生工作流协同方面仍存在割裂。

随着大语言模型、[[医疗AI Agent]]、[[RAG]]、[[Tool Calling]] 和 [[Workflow Orchestration]] 等技术发展，AI 已经具备一定的信息整理、上下文理解、工具调用和任务协同能力。但医疗场景具有高风险、高责任和强合规要求，AI 不应脱离医生独立作出诊疗决策，更不能替代医生。因此，本课题不以“自动诊断系统”为目标，而是研究一种面向医生工作流程的 [[医生AI Copilot]]：由医生主导，AI 负责辅助信息整理、流程调度、任务跟踪和知识增强。

本课题的核心问题是：如何通过 [[Agentic Workflow]] 将诊前、诊中、诊后任务组织为可控、可追溯、可审核的连续照护闭环，让 AI 真正嵌入医生工作流，而不是成为孤立的问答工具。

## 三、研究意义

### 1. 医疗服务流程意义

本课题关注诊疗全过程，而不是单点 AI 能力展示。通过诊前信息采集、诊中辅助决策和诊后医嘱执行三个 Workflow，可以减少医生在信息查找、资料整理和任务跟踪上的重复劳动，使医生更集中于临床判断和医患沟通。

### 2. 连续照护意义

诊疗服务并不止于一次问诊。诊后用药、康复、复查、复诊和随访反馈会直接影响后续诊疗质量。本课题通过 [[连续照护闭环]] 将诊后信息回流到下一次诊前和诊中流程，提升医疗服务的连续性。

### 3. 医疗AI安全意义

医疗 AI 应用必须坚持医生主导、人工审核、隐私保护和责任可追溯。本课题采用 Workflow 约束 Agent 行为，通过 [[Human-in-the-loop]]、[[医疗隐私与最小必要原则]] 和评估指标设计，避免完全自主 Agent 在医疗场景中带来的不可控风险。

### 4. 工程实践意义

本课题以模拟医疗数据和可演示 Workflow 为基础，探索医疗 Copilot Agent 的系统架构、数据流、工具调用和评估路径，为后续接入真实医院信息系统或参赛作品演示提供可复用方案。

## 四、国内外研究与应用现状

### 1. Agentic Workflow 与智能体工程

Anthropic 在 agentic system 设计中区分了预定义流程编排的 workflow 与由模型动态决策的 agent，强调应优先采用简单、可控、可解释的模式。OpenAI 的 agent 构建指南提出 Agent 可由模型、工具和指令构成，并通过工具、编排和护栏实现任务执行。LangGraph 等框架则将 Agent 工作流建模为状态、节点和边组成的图结构，支持中断、回退和人在环干预。

这些研究说明，面向医疗场景时，完全开放的自主 Agent 并不是最稳妥选择。将 Agent 能力放入可定义、可审计的 Workflow 中，更符合医疗安全和流程治理要求。

### 2. 医疗AI与临床决策支持

临床决策支持系统长期用于帮助医生理解病历、检查结果和诊疗知识。近年来，大语言模型和医学基础模型进一步提升了医学文本理解、对话式病史采集、知识检索和病历摘要能力。但相关研究也普遍指出，LLM 在医学场景中仍存在幻觉、知识时效、偏倚、责任归属和真实临床验证不足等问题。

因此，本课题将 AI 定位为“辅助医生的信息与流程协同者”，而不是直接输出最终诊疗结论的自动决策者。

### 3. 医疗数据互操作与系统集成

FHIR、SMART App Launch、CDS Hooks 等标准为医疗数据互操作、第三方应用授权和临床工作流触发提供了重要参考。对于本课题而言，这些标准说明医疗 Copilot 不必重建 HIS、EMR、LIS、PACS 等系统，而应通过标准接口或模拟接口调用所需数据，并把结果嵌入医生已有工作流。

### 4. 政策与伦理约束

WHO 医疗 AI 伦理治理指南强调保护人类自主性、促进安全、透明可解释、责任问责、公平包容和可持续治理。我国《互联网诊疗监管细则（试行）》明确要求人工智能软件不得冒用、替代医师本人提供诊疗服务，且不得自动生成处方替代医生行为。《卫生健康行业人工智能应用场景参考指引》也将医疗 AI 定位在辅助应用与场景创新层面。

因此，本课题的合规定位是：AI 辅助医生，不替代医生；AI 生成草稿和建议，不直接形成最终医疗决策；医生始终是审核者、采纳者和责任主体。

## 五、研究目标

本课题目标是设计并验证一个面向医生工作流程的医疗 Copilot Agentic Workflow 方案，形成贯穿诊前、诊中、诊后的连续照护闭环。

具体目标包括：

- 构建以 [[医疗Copilot Agent]] 为智能调度中枢的总体架构。
- 设计 [[诊前信息采集 Workflow]]、[[诊中辅助决策 Workflow]]、[[诊后医嘱执行 Workflow]] 三个核心流程。
- 明确患者信息、病历数据、检查结果、医疗知识和诊后反馈之间的 [[数据流设计]]。
- 通过 [[模拟数据设计]] 验证系统流程，不使用真实患者隐私数据。
- 建立覆盖技术价值、医疗价值和安全合规的 [[评估指标]]。
- 形成可用于比赛申报、方案展示和后续原型开发的开题报告与技术方案基础。

## 六、研究内容

### 1. 总体架构设计

系统以医疗 Copilot Agent 为核心，由三类能力支撑：

- 上下文能力：维护患者诊前、诊中、诊后信息。
- 工具能力：调用模拟 HIS、EMR、LIS、PACS、提醒和随访工具。
- 编排能力：根据医生工作阶段调度不同 Workflow。

### 2. 诊前信息采集 Workflow

诊前 Workflow 面向患者就诊前的信息准备，主要完成：

- 收集患者基础信息、主诉、症状描述和历史资料。
- 将自由文本整理为结构化诊前摘要。
- 识别缺失信息并生成补充问题。
- 汇总近期健康数据和历史诊疗记录。
- 为诊中辅助决策提供患者上下文。

### 3. 诊中辅助决策 Workflow

诊中 Workflow 面向医生接诊过程，主要完成：

- 整合诊前摘要、病历和检查结果。
- 辅助查询医疗知识、指南或相似管理建议。
- 生成病历摘要、诊疗要点或医嘱草稿。
- 标记不确定信息并提示医生确认。
- 将医生确认后的医嘱和管理任务传递给诊后流程。

### 4. 诊后医嘱执行 Workflow

诊后 Workflow 面向患者离开诊室后的持续管理，主要完成：

- 将医生确认的医嘱拆解为用药、康复、复查、复诊和随访任务。
- 生成患者侧提醒和执行计划。
- 记录患者反馈和健康数据。
- 标记未完成任务、异常情况和风险信号。
- 将诊后执行结果回流到下一次诊前信息采集。

### 5. 数据与知识支撑

本课题的数据来源采用模拟数据，不直接处理真实患者隐私数据。模拟数据包括患者基础信息、病历摘要、检查结果、医嘱任务、用药反馈、随访记录和风险等级等。

知识支撑包括医疗知识库、公开指南资料、项目 Wiki 节点和参考资料清单。RAG 能力用于知识查询和上下文增强，但输出必须保留来源、限制和不确定性说明。

### 6. 安全与人在环机制

系统设计必须遵循：

- 医生审核所有诊疗相关输出。
- AI 只生成辅助摘要、草稿、提醒和风险标记。
- 处方、诊断、医嘱调整等关键动作不得由 AI 自动完成。
- 患者数据遵循最小必要原则。
- 所有关键任务保留输入、输出、工具调用和医生确认记录。

## 七、技术路线

1. 梳理医生诊前、诊中、诊后的核心任务和信息流。
2. 建立医疗 Copilot Agent 总体架构，明确 Agent、Workflow、工具和数据源关系。
3. 设计三大 Workflow 的输入、处理步骤、输出和医生审核节点。
4. 构造模拟患者数据、模拟病历数据和模拟医嘱执行数据。
5. 实现或描述 Agent 对 RAG、Memory、Tool Calling 和 Workflow Orchestration 的调用方式。
6. 设计评估指标，从技术可用性、医疗流程价值、安全合规三个维度验证方案。
7. 形成开题报告、技术方案和演示材料。

## 八、可行性分析

### 1. 业务可行性

本课题聚焦医生真实工作流程中的信息整理、辅助查询和任务跟踪问题，需求明确，场景连续，适合通过 Workflow 方式拆解验证。

### 2. 技术可行性

当前大语言模型已具备文本摘要、结构化整理、工具调用、知识检索和多轮上下文维护能力。通过 Agentic Workflow 约束执行路径，可以降低完全自主 Agent 的不可控风险。

### 3. 数据可行性

本课题初期采用模拟数据验证流程，不依赖真实医院数据接入，能够规避隐私和合规风险。后续若进入真实场景，可通过 FHIR、SMART App Launch、CDS Hooks 或院内接口扩展。

### 4. 合规可行性

课题定位为医生辅助系统，不替代医生诊疗行为，不自动生成最终处方或诊断结论，符合医疗 AI 应用中“辅助、审核、可追溯”的基本要求。

## 九、创新点

### 1. 从单点问答转向连续照护闭环

本课题不是构建单一医学问答工具，而是将 AI 放入诊前、诊中、诊后连续流程中，使信息、任务和反馈形成闭环。

### 2. 用 Workflow 约束医疗 Agent

本课题强调 Agentic Workflow，而不是完全自主 Agent。通过预定义流程、医生审核节点和工具边界，使 AI 行为更加可控。

### 3. 强调诊后医嘱执行与反馈回流

许多医疗 AI 应用集中在诊中问答和病历生成，本课题将诊后医嘱执行、患者反馈和复诊准备纳入核心流程。

### 4. 面向比赛与原型验证的知识库化方案

课题已经在知识库中形成核心定义、Wiki 节点、知识总览、知识索引和参考资料清单，便于持续迭代、答辩展示和工程落地。

## 十二、风险与应对

|风险|表现|应对措施|
|---|---|---|
|AI 输出不准确|摘要遗漏、知识引用错误、建议不可靠|保留医生审核节点，标记不确定性，建立可追溯记录|
|医疗责任边界不清|用户误以为 AI 可替代医生|界面和流程中明确 AI 辅助定位，关键动作由医生确认|
|隐私与数据合规风险|真实患者数据泄露或越权使用|初期使用模拟数据，后续遵循最小必要原则和权限控制|
|Workflow 过度复杂|系统难以实现或演示重点分散|优先实现三条主流程，避免无关功能扩张|
|评价指标不清|难以证明项目价值|从技术、医疗价值、安全三个维度建立可量化指标|
