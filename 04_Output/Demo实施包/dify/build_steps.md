# Dify 搭建步骤

## 目标

搭建一个可演示的医生 AI Copilot Demo：1 名高血压复诊模拟患者完成一次完整医疗旅程。重点不是展示三个功能，而是展示 Agent 如何读取 Patient Memory、维护风险画像，并把诊后反馈回流到下一次诊前。

## 应用结构

推荐先建 1 个 Chatflow，内部按四段节点组织：

1. Agent 上下文装载：读取患者档案、Patient Memory、风险画像。
2. 诊前信息采集：把连续照护线索放入诊前摘要。
3. 诊中 RAG 辅助决策：结合风险画像生成供医生核对的管理方向。
4. 诊后医嘱执行与反馈回流：将患者执行反馈写回 Memory，并更新风险画像。

如果 Dify 的跨阶段状态传递不顺，立即改成 3 个独立应用：

- App 1：诊前摘要生成。
- App 2：诊中辅助建议与医嘱草稿。
- App 3：诊后任务跟踪与反馈回流。

演示时人工复制上一阶段“医生确认后的输出”到下一阶段输入，这仍然符合 HITL 设计。

## 配置步骤

1. 创建知识库。
   - 名称：`高血压复诊 Demo 知识库`
   - 导入：`../data/knowledge_hypertension_demo.md`
   - 分段方式：按 Markdown 标题切分。

2. 创建应用。
   - 名称：`医生 AI Copilot - 高血压复诊闭环 Demo`
   - 模式：Chatflow 或 Workflow。
   - 模型：优先 DeepSeek Chat；备选通义千问或其他稳定中文模型。

3. 配置全局提示词。
   - 使用 `prompts.md` 的“全局系统提示词”。

4. 配置诊前节点。
   - 输入患者文本、`patient_wang_001.json`、`patient_memory_wang_001.json` 和 `risk_profile_wang_001.json`。
   - 输出结构化诊前摘要。
   - 输出顶部必须展示 Patient Memory 中的连续照护线索。
   - 输出必须包含风险画像提示。
   - 在输出末尾保留医生确认区。

5. 配置诊中节点。
   - 输入医生确认后的诊前摘要。
   - 输入 Patient Memory 和风险画像。
   - 触发知识库检索。
   - 输出带来源的辅助建议和医嘱草稿。
   - 输出“建议写回 Patient Memory 的确认项”。
   - 医嘱草稿不得直接进入诊后，必须等待医生确认。

6. 配置诊后节点。
   - 输入医生确认后的医嘱。
   - 使用 `followup_tasks.jsonl` 展示任务拆解。
   - 使用 `followup_events.jsonl` 展示 7 天反馈。
   - 输出异常标记、风险画像更新草稿、Patient Memory 写回内容和下一次诊前反馈摘要。

7. 配置追溯展示。
   - 每阶段保留输入、AI 输出、医生确认文本。
   - 展示 Patient Memory 从“初始病例”到“诊后回流”的变化。
   - 展示风险画像从诊前到诊后的变化。
   - 答辩时强调：关键输出均可审核、可修改、可拒绝。

## 本地兜底

若 Dify 现场无法访问：

1. 打开本实施包中的数据文件和提示词文件。
2. 按 `演示脚本.md` 讲解三段 Workflow。
3. 播放提前录制好的 Dify 全流程视频。
4. 展示 `tool_schemas.json` 证明工具接口已冻结。

## 实施检查

- 诊前输出不能出现诊断或治疗方案。
- 诊前必须读取 Patient Memory 和风险画像。
- 诊中输出必须有来源片段。
- 医嘱草稿必须有医生确认区。
- 诊后输出不能自动改药。
- 回流摘要必须能作为下一次诊前输入，并作为演示高潮呈现。
