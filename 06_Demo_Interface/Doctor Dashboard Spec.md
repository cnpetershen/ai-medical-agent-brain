# Doctor Dashboard Spec

本文定义 Demo Presentation Layer 的医生展示规格。本文只描述展示结构，不开发前端，不修改 Runtime，不设计聊天机器人界面。

## 设计目标

Doctor Dashboard 用于把 Runtime 输出转换为医生可理解、可审核、可追溯的展示。

展示层必须体现三条主线：

- AI 辅助：AI 只负责整理信息、检索依据、生成草稿和汇总反馈。
- 医生确认：所有关键诊疗相关输出都要展示确认状态。
- 连续照护闭环：诊后反馈必须回流为下一次诊前的重要上下文。

## 整体信息架构

Dashboard 采用三阶段工作台结构，不采用聊天窗口，不以自由对话作为主界面。

主视图分为三个阶段：

1. 诊前
2. 诊中
3. 诊后

每个阶段都由四类区域构成：

- 患者上下文区
- AI 输出区
- 医生确认区
- 追溯与状态区

## 一、诊前工作台

### 展示目标

让医生在接诊前快速理解患者背景、近期风险、AI 整理结果以及仍需确认的信息。

### 必须展示内容

#### 1. 患者摘要

来源：

- `patient_wang_001.json`
- `pre_visit` Runtime 输出
- `patient_memory_wang_001.json`

展示字段：

- 患者别名、年龄、性别
- 就诊类型
- 本次主诉
- 既往高血压病史
- 当前用药
- 近期家庭血压趋势
- 上一次诊后回流线索

展示形式建议：

- 左侧患者卡片
- 中间结构化摘要
- 顶部标记“诊前阶段”

#### 2. 风险画像

来源：

- `risk_profile_wang_001.json`

展示字段：

- 总体风险提示
- 血压控制风险
- 用药依从性风险
- 症状变化风险
- 生活方式风险
- 连续照护断点风险

展示形式建议：

- 风险优先级列表
- 每个风险维度附“医生关注动作”

#### 3. AI整理结果

来源：

- `pre_visit` 中 `identify_key_and_missing_information`
- `pre_visit` 中 `generate_structured_pre_visit_summary`

展示字段：

- 关键事实
- 风险线索
- AI 诊前摘要草稿
- 近期血压趋势整理

展示约束：

- 必须标记为“AI整理结果”或“辅助草稿”
- 不得显示为最终诊断或治疗结论

#### 4. 待确认信息

来源：

- `pre_visit` 中缺失信息问题
- HUMAN 确认前状态

展示字段：

- 头晕的时间、频率、持续时长
- 是否伴随胸闷、胸痛、视物模糊
- 家庭血压测量方式是否规范

展示形式建议：

- 待确认问题列表
- 每项问题可显示状态：`待确认` / `已确认`

### 医生确认区

展示字段：

- 当前确认状态
- 医生确认 ID
- 是否已写入 Memory

状态定义：

- `approve`
- `modify`
- `reject`

## 二、诊中工作台

### 展示目标

让医生在诊中看到知识依据、AI 辅助建议、医嘱草稿以及当前确认状态。

### 必须展示内容

#### 1. RAG参考依据

来源：

- `during_visit` 中 `retrieve_medical_knowledge`
- `knowledge_hypertension_demo.md`

展示字段：

- 片段标题
- 来源
- 要点摘要

展示约束：

- 每条辅助建议旁必须可回看来源
- 若检索不到内容，必须展示“未检索到充分来源”

#### 2. 辅助建议

来源：

- `during_visit` 中 `organize_clinical_context`

展示字段：

- 诊中关键上下文
- 医生可核对的管理方向
- 安全提示

展示约束：

- 明确为“辅助建议草稿”
- 不得展示为 AI 最终判断

#### 3. 医嘱草稿

来源：

- `during_visit` 中 `generate_order_draft`

展示字段：

- 管理计划草稿
- 医生辅助核对点
- 医疗决策状态：`待医生确认`

展示约束：

- 必须标记“待医生确认”
- 不得直接进入“已生效医嘱”

#### 4. 医生确认状态

来源：

- `during_visit` HUMAN Node 输出
- `in_visit_memory`

展示字段：

- 确认结果：`approve` / `modify` / `reject`
- 医生确认 ID
- 是否已写入诊中 Memory

### 展示重点

诊中工作台不是让医生“跟 AI 聊天”，而是让医生查看：

- 诊前已经确认的上下文
- 知识依据
- AI 整理出的可核对草稿
- 医生是否已确认

## 三、诊后工作台

### 展示目标

让医生看到随访任务、患者反馈事件、Memory 变化，以及下一次诊前将会继承的上下文。

### 必须展示内容

#### 1. 随访任务

来源：

- `followup_tasks.jsonl`
- `post_visit` 中 `create_followup_tasks`

展示字段：

- 任务标题
- 类型
- 执行频率
- 当前状态
- 来源：医生确认后的医嘱

展示约束：

- 明确为任务管理，不是自动医疗行为

#### 2. 患者反馈事件

来源：

- `followup_events.jsonl`
- `record_patient_feedback`
- `get_followup_status`

展示字段：

- 日期
- 事件类型
- 反馈内容
- agent flag
- 是否需要医生复核

重点事件：

- 第 3 天头晕较前明显
- 第 4 天漏服
- 第 7 天随访摘要

#### 3. Memory变化

来源：

- `post_visit` Runtime 输出中的 Memory before / after
- `patient_memory_wang_001.json`

展示字段：

- 写回前 Memory
- 写回后 Memory
- 新增异常事件
- 新增诊后依从性摘要
- 新增下一次诊前摘要

展示形式建议：

- before / after 对照区
- 仅高亮变化字段，不展示整份原始 JSON

#### 4. 下一次诊前上下文

来源：

- `post_visit_memory.next_previsit_summary`

展示字段：

- 7 天内完成多数反馈记录
- 出现 1 次漏服
- 第 3 天头晕较前明显，后续减轻
- 家庭血压多数仍偏高

展示意义：

- 这是连续照护闭环的核心展示点
- 必须放在诊后工作台的高优先级位置

### 医生确认区

展示字段：

- 诊后确认结果
- 医生确认 ID
- 是否已写回 Memory

## 跨阶段共通元素

每个阶段都应固定展示以下状态标签：

- `AI辅助`
- `待医生确认` / `已医生确认`
- `已写入Memory` / `未写入Memory`
- `来源可追溯`

## 不应设计的界面形式

- 不设计聊天机器人对话框作为主界面
- 不设计“AI 自动诊断结果”卡片
- 不设计“AI 自动开方”操作区
- 不设计“自动治疗方案调整”视图

## 展示层核心结论

Doctor Dashboard 的本质不是一个聊天 UI，而是一个医生工作台。它把 Runtime 中的节点执行结果、知识依据、医生确认状态和 Memory 回流结果组织成三个阶段的连续照护视图。
